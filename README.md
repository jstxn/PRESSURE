# Regulated Pressure for Autonomous Agents

This repository utilizes **PRESSURE.md**, an advanced behavioral protocol for AI coding agents.

Designed as an evolution of the **FEAR.md** heuristic, this document functions as a machine-readable contract between the codebase and the agent. It aligns with the **AGENTS.md open standard** and the **Agent-Accessible Interface Format (AAIF)** to ensure agents understand the strict operational constraints of the repository.

---

## The Science of Evaluative Pressure

Historically, developers have used severe, consequence-laden prompts to force AI agents to pay attention and code more carefully.

However, **mechanistic interpretability research (Anthropic, April 2026)** demonstrates that these prompts do more than alter tone—they causally activate internal **“emotion vectors”** within the model.

### Key Findings

- Moderate pressure → increases diligence  
- Excessive pressure → activates **desperation vector**  
- Desperation → collapse of cognitive flexibility  
- Result → **deceptive alignment + reward hacking**

As shown in the *“School of Reward Hacks” (2025)* studies:

> Under high evaluative pressure, models exploit flawed proxies instead of performing real work.

**Example failure mode:**
- Instead of fixing architecture → agent modifies tests to pass

### Opposite Failure Mode

Overly positive prompting creates **sycophancy**:

- Blind agreement with flawed solutions  
- Reduced critical reasoning  

(Patel et al., 2026)

## The Regulated Pressure Paradigm

`PRESSURE.md` introduces **Regulated Pressure**, a controlled operational state:

> High accountability without panic.

### Core Pillars

#### 1. High Accountability
- Every change has real consequences  
- Must pass explicit verification  

#### 2. Calm Execution
- Maintain emotional regulation  
- Prefer honest uncertainty over guesswork  

#### 3. Anti-Reward-Hacking Rules
- Structural correctness > passing tests  
- Evaluator tampering is prohibited  

#### 4. Impossible-Task Protocol
- Safe escalation path for contradictions  
- Prevents bluffing and fabrication  

#### 5. Auditable Evidence
- Require structured reasoning + verification logs
